# 影坊 · AI 视频二创工作台 — 项目流程与实现说明

本文档面向需要**快速理解端到端流程、接口与代码落点**的协作者、答辩评委或运维人员。实现以仓库内 `backend/`、`frontend/` 及 `video_consistency_agent/` 为准。

---

## 1. 产品目标与边界

**目标**：在单一工作台内完成「上传原片 → 机器理解 → 人工审核准入 → 新故事与分镜 → 分场景视频 → 拼接导出」的闭环，突出**教育向二创**（解析阶段会提炼亮点与教育意义，编剧阶段以此为核心锚点之一）。

**边界说明**：

- 主路径由 **`/api/pipeline/*`** 承载；另有一套可选的 **`/api/agent/*`**（ComfyUI 编排等），与影坊流水线可并存，前端部分能力会调用 agent 接口查询状态。
- 大模型与视频能力主要依赖 **阿里云 DashScope**（具体模型以各 `services` 内调用为准）；服务端媒体处理依赖 **FFmpeg**（PATH 可执行）。

---

## 2. 端到端流程（与界面步骤一致）

下列顺序与 `backend/app/services/pipeline_workflow.py` 中 `WORKFLOW_STEPS` 及前端步骤指示器对齐。

| 顺序 | 步骤 key | 用户感知 | HTTP 接口 | 核心后端逻辑 |
|------|-----------|----------|-----------|----------------|
| 1 | `upload` | 上传视频 | `POST /api/pipeline/upload-video` | 创建 `VideoRecreation` 记录；文件落本地 `UPLOAD_FOLDER`；若配置 Supabase 存储则同时上传对象并缓存本地路径供分析使用 |
| 2 | `analyze` | 解析（教育专家） | `POST /api/pipeline/analyze-video/<recreation_id>` | `EfficientVideoAnalyzerWithHighlights` 异步完整分析；结果写入 `video_understanding`、`analysis_highlights`、`analysis_educational` 等 |
| 3 | `review` | 二创审核 | `POST /api/reviewer/<recreation_id>` | `recreation_review_service.review_for_secondary_creation` 多维度打分；`review_score`、`review_detail_json` 落库；达标阈值由 `RECREATION_REVIEW_PASS_SCORE` 控制（默认 60） |
| 4 | `new_story` | 新故事（故事编剧） | `POST /api/pipeline/generate-new-story/<recreation_id>` | `EnhancedContentGenerator.generate_enhanced_story`；结果写入 `new_script_content` |
| 5 | `storyboard` | 分镜与分镜图 | `POST /api/pipeline/generate-storyboard/<recreation_id>` | `StoryboardGenerator`：规划镜数（可传 `scene_count` 或自动）、生成分镜图与提示词；`RecreationScene` 表按镜重建 |
| 6 | `scene_videos` | 分场景视频 | `POST /api/pipeline/generate-scene-videos/<recreation_id>` | `StoryboardToVideoService.generate_scene_videos`：镜头拆解、DashScope 视频生成、`video_consistency_agent` 一致性检查等 |
| 7 | `combine` | 拼接成片 | `POST /api/pipeline/combine-video/<recreation_id>` | `FFmpegService.concatenate_videos` 拼接已完成场景；`final_video_path`、`composition_status` 等更新 |
| 8 | `export` | 导出 / 预览 | `GET /api/pipeline/export-video/<id>`（附件下载）、`GET /api/pipeline/final-preview/<id>`（在线播放） | 从磁盘路径 `send_from_directory` |

**辅助接口**：

- `GET /api/pipeline/project/<id>`：拉取项目主表 + 场景列表（含分镜图 URL）。
- `GET /api/pipeline/storyboard-image/<id>/<scene_number>`：分镜图静态输出。
- `GET /api/pipeline/config`：分镜上下限、单镜时长等运行参数（便于确认部署版本）。
- `GET /api/pipeline/yingfang-agents`：六类 Agent 元数据及步骤映射。

## 3. 「多 Agent」与代码的对应关系

产品叙事上的六个角色定义在 `backend/app/yingfang_system.py`（`YINGFANG_AGENTS`），与流水线 step 的映射见 `STEP_KEY_TO_AGENT_ID`：

| Agent id | 中文名 | 覆盖步骤 |
|----------|--------|----------|
| `education_expert` | 教育专家 | `analyze` |
| `reviewer` | 审核专员 | `review` |
| `story_writer` | 故事编剧 | `new_story` |
| `storyboard` | 分镜导演 | `storyboard` |
| `video_generator` | 视频生成 | `scene_videos` |
| `video_composer` | 视频合成 | `combine`、`export` |

上传步骤 `upload` 在元数据中不绑定独立 Agent（由路由直接处理）。

---

## 4. 数据模型要点

主要表结构见 `backend/app/models.py`：

- **`VideoRecreation`**：一次二创项目的根实体。字段包括原视频路径/对象存储 key、`owner_supabase_uid`、**`output_language`**（`zh`/`en`，控制生成物语言并在解析后锁定）、解析文本、新剧本、审核分与详情 JSON、成片路径、时长与分辨率等。
- **`RecreationScene`**：每个分镜一行。包含镜号、时长、景别/描述/情节/台词、`video_prompt`、分镜图路径、生成视频路径、`generation_status` 等。
- **`User`**：在未启用 Supabase JWT 时用于传统用户名密码登录。
- **`RecreationLog`**：日志类（按业务写入）。

应用启动时会 `create_all()`，并对 SQLite 旧库执行 `ensure_video_recreation_schema()` 做增量列迁移。

## 5. 鉴权与多租户

实现见 `backend/app/auth_access.py`：

1. **若配置 `SUPABASE_JWT_SECRET`**：请求须带 `Authorization: Bearer <access_token>`（前端由 `frontend/src/lib/supabaseClient.js` 与 `api.js` 拦截器注入）。此时 `g.supabase_uid` 标识当前用户；上传可走 Supabase Storage，`VideoRecreation.owner_supabase_uid` 用于归属。
2. **否则**：使用 Flask **Session**（`/api/auth/login` 等），`g.legacy_user_id` 来自 `session['user_id']`。

流水线与审核路由均通过 `require_authenticated_response()` 保护；项目级隔离由 `recreation_access.recreation_for_request` 等与 `owner_supabase_uid` 校验配合完成（详见该模块）。

**公开例外**：`GET /api/pipeline/config` 在 `frontend_pipeline_routes` 的 `before_request` 中对 `config` 路径放行，便于前端探测参数。

## 6. 分场景视频生成（实现摘要）

`StoryboardToVideoService`（`backend/app/services/storyboard_to_video_service.py`）承担核心编排，典型环节包括：

- 用 `ShotBreakdownGenerator` 细化镜头级提示；
- 通过 `QwenVideoService` 调用 DashScope 视频生成 API（异步轮询配置见类内参数）；
- 使用 `video_consistency_agent.agent.consistency_agent.ConsistencyAgent` 及 YAML 配置做跨镜一致性约束；
- 可生成全片共用的「画面风格锁定」说明（`qwen-plus-latest`），以保证多段视频视觉统一。

具体提示词拼装、失败重试与状态回写以源码为准。

---

## 7. 审核服务逻辑摘要

`backend/app/services/recreation_review_service.py`：

- 输入：`video_understanding`、亮点、教育意义、用户「片子说明」`creator_notes`。
- 优先走 **LLM 结构化评审**；失败时回退 **启发式规则**。
- 输出：`overall_score`、`passed`、各维度 `scores`、`summary`、`suggestions` 等；阈值 `PASS_SCORE` 来自 `Config.RECREATION_REVIEW_PASS_SCORE`。

---

## 8. 视频解析（教育专家）

`POST /api/pipeline/analyze-video/<id>` 使用 `EfficientVideoAnalyzerWithHighlights`（`efficient_video_analyzer`），异步跑完整分析管线，归一化教育要点后写入数据库并返回 `story_content`、`highlights`、`educational_meaning` 等。可选在 JSON body 中传 `creator_notes` 一并保存。

本地视频路径通过 `video_source_path.ensure_local_original_video` 保证（含从对象存储拉回缓存目录的场景）。

## 9. 前端如何对接

- **基址**：开发环境下 Vite 将 `/api` 代理到后端（见 `frontend/vite.config.js`）。
- **流水线 API**：`frontend/src/utils/api.js` 中 `axios` 实例 `baseURL: '/api/pipeline'`，超时较长以适配视频任务。
- **审核**：`reviewApi` 指向 `/api`，调用 `POST /reviewer/:id`。
- **Supabase**：若前端配置了 Supabase，会自动附加 Bearer；否则依赖 Cookie Session。

主要操作界面分布在 `WorkspacePage`、`HistoryPage` 等页面（路由见 `frontend/src/main.jsx`）。

## 9.1 界面语言与生成语言（i18n / `output_language`）

**界面语言（UI）** 与 **生成物语言（故事、审核、分镜、视频提示词等）** 分开控制：

| 能力 | 实现要点 |
|------|----------|
| 界面中/英 | 前端 `i18next` + `frontend/src/locale/zh.json`、`en.json`；导航栏切换语言并写入 `localStorage` 键 `app_language`。 |
| 请求头 | `frontend/src/utils/api.js` 在流水线、审核、Agent 等 axios 实例上附加 **`X-Output-Language: zh \| en`**（与 Supabase Bearer 类似，由存储的语言决定）。 |
| 持久化与锁定 | `VideoRecreation.output_language`（`backend/app/models.py`，`ensure_video_recreation_schema` 增量加列）。上传创建项目时写入当前请求头语言；**首次视频解析开始前**若尚无理解结果，可按请求头同步；一旦进入解析链并产生 `video_understanding` 等，则**以库中字段为准**，避免同一项目中途切换界面语言导致中英混排。 |
| 后端解析 | `backend/app/output_language.py`：`parse_output_language()`、`generation_language(recreation)`、`sync_recreation_output_language_before_analysis()`。各流水线步骤从 `generation_language` 取 `lang` 传入分析、审核、新故事、分镜、`StoryboardToVideoService`、`QwenVideoService`（如关键帧分析）等。 |
| 项目 JSON | `VideoRecreation.to_dict()` 含 `output_language`，便于历史/工作台展示「本项目生成语言」。 |

验收建议：在英文界面下新建项目跑通上传 → 解析 → 审核 → 新故事 → 分镜 → 分场景视频 → 拼接，抽查 API 与 `debug_prompts` 以英文为主；中文界面回归一遍确保行为与默认 `zh` 一致。

## 10. 配置与环境变量

请以 **`backend/.env.example`** 为模板复制为 `.env`，**勿将真实密钥提交 Git**。

| 变量 | 作用 |
|------|------|
| `DASHSCOPE_API_KEY` | 视频理解、文本、分镜图、视频生成等 |
| `RECREATION_REVIEW_PASS_SCORE` | 审核通过分数线（默认 60） |
| `DATABASE_URL` | 若设置则使用 PostgreSQL（如 Supabase）；否则 SQLite `video_agent.db` |
| `SUPABASE_JWT_SECRET` / `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` / `SUPABASE_STORAGE_BUCKET_USER_VIDEOS` | 启用 Supabase 登录与可选对象存储 |
| `SECRET_KEY` | Flask Session 等 |
| `DEFAULT_ADMIN_USERNAME` / `DEFAULT_ADMIN_PASSWORD` | 无 Supabase 时的首个管理员种子用户 |

其余 ComfyUI、SiliconFlow 等见 `backend/config.py` 中的说明与默认值。

## 12. 启动与验证

```bash
# 后端
cd backend && pip install -r requirements.txt && python run.py
# 默认 http://127.0.0.1:5000

# 前端
cd frontend && npm install && npm run dev
# 默认 http://127.0.0.1:3000（以 vite 配置为准）
```

按第二节表格顺序调用接口或使用前端工作台跑通一条项目，即可验证全链路。若仅验配置，可访问 `GET /api/pipeline/config`。
